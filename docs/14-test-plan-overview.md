# 全功能测试总览与分工

这组测试文档用于把 `开放吞噬 / agar.io-clone` 当前功能完整测一轮，并把任务拆给真人测试员和 AI/自动化测试。

## 文档分工

- [15-test-plan-human-checklist.md](15-test-plan-human-checklist.md)：给真人测试员。重点看手感、理解成本、UI、玩法反馈、NPC 情绪、移动端体验。
- [16-test-plan-ai-checklist.md](16-test-plan-ai-checklist.md)：给 AI 或脚本。重点跑构建、Socket、回归、多客户端、异常路径、日志和部署验收。
- 本文档：给测试负责人。用于安排环境、轮次、记录格式和通过标准。

## 测试标签

- `真人优先`：必须由人确认，不能只靠脚本替代。
- `AI优先`：适合 Playwright、脚本、多客户端、日志断言或重复回归。
- `混合`：AI 先验证状态/事件，真人再判断体验是否自然。

## 环境

- 本地地址：`http://127.0.0.1:3000`
- 可选线上地址：`http://124.222.83.113:3000/`
- 不要把 `http://124.222.83.113/` 当成这个项目，80 端口是另一个 `food` 服务。

## 启动命令

```bash
npm install
npm run build
npm start
```

历史回响 / 调试 / 快速结算建议另开一轮：

```bash
V5_DEMO_MODE=1 GHOST_DEBUG=1 V5_GHOST_TRACE_RECORDING=1 npm start
```

如需验证持久历史回响：

```bash
V5_DEMO_MODE=1 GHOST_DEBUG=1 V5_GHOST_TRACE_RECORDING=1 V5_GHOST_PERSISTENCE=1 npm start
```

关闭 NPC 回归基础游戏：

```bash
V5_NPC_ENABLED=0 npm start
```

## 设备矩阵

- 桌面 Chrome：主测。
- 桌面 Safari 或 Firefox：兼容性抽测。
- 手机浏览器或移动模拟器：移动按钮、触摸移动、布局抽测。
- 至少 2 个浏览器窗口：多人、吞噬、连接、聊天、排行榜必须用。
- 压力/回归：5-10 个自动化客户端。

## 推荐执行顺序

1. AI 先跑 P0 冒烟：构建、首页、Socket、控制台错误、多客户端基础路径。
2. 真人第一轮：首屏、身体签名、名片、入场、移动、吃食物、聊天、NPC 基础感受。
3. 双人真人：吞噬、连接、关系、名片同步、死亡结算。
4. AI 定向制造：部位掉落、回响、快速结算、管理员、异常 payload、刷新压力。
5. 真人第二轮：手机、深色模式、长时间玩 15 分钟，记录“哪里不顺、哪里看不懂、哪里不好玩”。
6. 如需上线：AI 跑部署验收，真人在公网完成一局冒烟。

## AI优先测什么

- 构建、启动、HTTP 可访问、静态资源加载。
- 单元测试、已有 demo 脚本、npm audit 结果收集。
- 昵称合法性、语言切换后关键文案存在。
- Socket 握手、移动心跳、聊天事件、排行榜事件。
- 多客户端加入/断开/观战/踢人。
- localStorage 持久化：语言、名片、签名、草稿。
- 回合超时、快速结算、死亡结算是否出现。
- 控制台错误、网络 404、服务端异常日志。
- 异常 payload、刷新压力、LLM/数据库不可用降级。

## 真人优先测什么

- 鼠标移动、方向键移动、分裂、喷射的手感。
- 吃食物、吃玩家、碰病毒、变大变慢是否“感觉对”。
- 名片画板是否好用，图层/锁定/草稿是否能被普通人理解。
- 身体签名面板的引导是否清楚，相似度结果是否可信。
- NPC 说话、跟宠、任务反馈、历史回响是否有存在感。
- UI 是否遮挡、拥挤、文字是否看得懂，手机上是否能玩。
- 一局结束后的结算信息是否能帮助玩家理解自己发生了什么。

## 混合项怎么测

- 连接/共鸣/断裂：AI 制造距离条件，真人看反馈是否明显。
- 身体部位系统：AI 断言 HUD 数字和同步字段，真人看是否知道自己捡到了什么。
- 玩家名片同步：AI 截图比对，真人看头像识别是否舒服。
- 历史回响：AI 检查 ghost payload，真人看是否像“上一局留下的痕迹”。
- NPC 任务奖励：AI 验证事件和部位 source，真人判断 NPC 文案和奖励出现是否自然。

## 通过标准

- P0/P1 问题全部关闭或有明确延期理由。
- 两份清单都至少完成一轮。
- 所有 `真人优先` 项必须有真人反馈。
- 所有 `AI优先` 项必须有命令、截图、日志或脚本结果。
- `混合` 项至少先有自动化/日志证据，再有人确认体验是否可接受。
